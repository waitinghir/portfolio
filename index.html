Project 01 — Gamification × AI

Lunar New Year Challenge新春大作戰
新春大作戰

By Tim 2025 7 min read閱讀 7 分鐘
▶ Play the Game▶ 開始遊戲 ⟨/⟩ GitHub

Overview

Lunar New Year Challenge (新春大作戰) is a three-stage mobile web game that transforms traditional Chinese New Year cultural elements — red envelopes, lottery tickets, and firecrackers — into playable game mechanics. The entire game runs as a single HTML file in any mobile browser, requiring zero installation. The goal was simple: build something fun enough to share with family and friends during the holiday season, while celebrating the cultural traditions that make Lunar New Year special.

Why I Built This

I wanted to explore whether a complete, polished mobile game could be built entirely through human-AI collaboration — with me providing the creative direction and gameplay feedback, and Claude handling all the technical implementation. Lunar New Year felt like the perfect theme: it's culturally rich, visually vibrant, and inherently social. Instead of just another greeting card app, I wanted to create something people would actually play.

The Three Stages

Each stage introduces a completely different gameplay type, keeping players engaged through variety rather than repetition.

Stage 1 — Red Envelope Defense: Players control a horse character (a nod to the Chinese zodiac) and catch red envelopes falling from the sky while dodging obstacles. It's a classic action format built around an HP system, collision detection, and touch controls — designed to feel snappy and satisfying on a phone screen.

Stage 2 — New Year Lottery: A memory-based challenge where players must remember and collect the correct number combinations. This one plays on the "scratch card" atmosphere of the holiday season, testing short-term memory instead of reflexes.

Stage 3 — Firecracker Frenzy: A survival mode where players dodge waves of firecrackers with varying trajectories and blast patterns. The longer you survive, the more intense it gets — multiple explosion types keep the visuals fresh even in extended sessions.

Between stages, animated transition screens and firework particle effects give the game a complete, polished feel — not just three mini-games stitched together, but a cohesive experience from start to finish.

How AI Was Used

This project was built entirely through conversation with Claude. The collaboration model worked like this: I described what I wanted a stage to feel like, Claude produced a fully functional implementation, I playtested it on my phone, and then I fed back specific observations — "the red envelopes fall too fast," "the win button is too easy to miss-tap," "the explosions feel repetitive after 30 seconds." Claude would then adjust parameters, restructure UI elements, or add visual variety based on that feedback.

What made this work wasn't just code generation — it was the iterative loop. Claude built the entire Canvas game engine from scratch, including the game loop, state machine, collision system, particle effects, and multi-layer canvas rendering. But the quality of the game came from dozens of test-and-refine cycles where I translated subjective gameplay feel into concrete change requests.

Key Design Decisions

Memory games need real memory pressure. The first version of Stage 2 displayed the target numbers on screen the entire time — which completely defeated the purpose. Removing the persistent hint was an obvious fix in hindsight, but it reinforced an important principle: the core tension of a memory game is "can you remember it?" Anything that undermines that tension undermines the entire mechanic.

Primary actions must look primary. After clearing a stage, the "Next Stage" and "Retry" buttons were originally the same size and prominence. During playtesting, I kept accidentally hitting "Retry" right after winning — turning a moment of triumph into frustration. The fix was straightforward: make "Next Stage" visually dominant and demote "Retry" to a secondary style. It's a basic UX principle, but one you only truly internalize when you feel the pain yourself.

Difficulty curves can't be designed on paper. Both Stage 1 and Stage 3 involve objects spawning continuously. Too fast and the player is overwhelmed instantly; too slow and it's boring. Finding the sweet spot — "consistently tense but never hopeless" — required playing the game over and over and nudging spawn rates by small increments. No amount of theoretical planning could have replaced that hands-on tuning.

Visual repetition kills spectacle. Stage 3's explosions initially used a single animation pattern. It looked impressive for the first few seconds, then became wallpaper. Adding multiple explosion styles and particle variations solved this — the same mechanic felt dramatically more engaging simply because each detonation looked slightly different.

Tech Stack

The game is a single HTML file using HTML5 Canvas for rendering, vanilla JavaScript for game logic and state management, CSS3 animations for UI transitions, the Touch Events API for mobile controls, multi-canvas layering for visual depth (firework effects rendered on separate layers), and a custom particle system for all explosion and celebration effects. No frameworks, no build tools, no dependencies — just one file you can open in a browser and play.

Reflections

The biggest takeaway from this project isn't technical — it's about the collaboration model. Building a game through conversation with an AI inverts the traditional workflow. Instead of writing code and then testing, I was playing and then describing. The hardest part was articulating "game feel" — an inherently fuzzy concept — precisely enough for Claude to translate it into parameter changes. Saying "it feels too hard" isn't useful. Saying "enemies spawn every 0.8 seconds and I can't physically move my thumb fast enough to dodge them all" is.

The cultural elements weren't just a skin. Red envelopes, lottery, and firecrackers each naturally map to a different game genre (action, puzzle, survival), which meant the theme actually drove gameplay variety rather than just decorating it. That was a happy discovery — the best design decisions often come from constraints rather than from open-ended brainstorming.

If I were to do it again, I'd add a scoring system that persists across all three stages and some form of social sharing — a screenshot or short animation players could send to friends. The game was built to be shared, and making that sharing frictionless would have completed the loop.

概覽

新春大作戰是一款三關式手機網頁遊戲，將農曆新年的傳統文化元素——紅包、樂透、爆竹——轉化為可遊玩的遊戲機制。整個遊戲以單一 HTML 檔案運行在任何手機瀏覽器上，零安裝。目標很簡單：做一個有趣到值得在過年期間和親友分享的東西，同時慶祝讓農曆新年特別的文化傳統。

為什麼做這個

我想驗證一個想法：一款完整、有品質的手機遊戲，能否完全透過人與 AI 的協作來完成——我負責創意方向和遊玩回饋，Claude 負責所有技術實作。農曆新年是完美的主題：文化底蘊豐富、視覺鮮豔、天然適合社交分享。與其做一張電子賀卡，我想做人們真正會玩的東西。

三個關卡

每一關都是完全不同的遊戲類型，透過多樣性保持玩家的投入感。

第一關——紅包防衛戰：玩家操控一匹馬角色（呼應生肖主題），接住從天而降的紅包，同時閃避障礙物。這是經典的動作遊戲模式，建立在血量系統、碰撞偵測和觸控操作之上——設計上追求在手機螢幕上的爽快手感。

第二關——新春大樂透：記憶型挑戰，玩家需要記住並收集正確的數字組合。這一關玩的是年節「刮刮樂」的氛圍，考驗的是短期記憶而非反射神經。

第三關——爆竹一聲響：生存模式，玩家閃避不同軌跡和爆炸範圍的爆竹浪潮。撐越久越激烈——多種爆炸類型讓視覺在長時間遊玩中保持新鮮。

在關卡之間，過場動畫搭配煙火粒子特效讓遊戲有完整的質感——不是三個小遊戲硬拼在一起，而是從頭到尾一氣呵成的體驗。

AI 如何參與

這個專案完全透過與 Claude 的對話來完成。協作模式是這樣的：我描述一個關卡要有什麼「感覺」，Claude 產出完整可運行的實作，我在手機上實測，然後回饋具體觀察——「紅包掉太快了」、「過關按鈕太容易按錯」、「爆炸看 30 秒就膩了」。Claude 再根據回饋調整參數、重新排列 UI 元素或增加視覺多樣性。

讓這一切運作的不只是程式碼生成——而是迭代循環。Claude 從零建立了整個 Canvas 遊戲引擎，包含遊戲迴圈、狀態機、碰撞系統、粒子特效和多層畫布渲染。但遊戲的品質來自數十次測試與調整的循環，在這些循環中，我把主觀的遊戲手感轉譯為具體的修改需求。

關鍵設計決策

記憶遊戲需要真正的記憶壓力。第二關的第一版在畫面上全程顯示目標數字——這完全消除了挑戰性。移除持續提示事後看來理所當然，但它強化了一個重要原則：記憶遊戲的核心張力在於「你記不記得住」，任何削弱這個張力的設計都會讓整個機制失去意義。

主要動作必須看起來就是主要的。過關後，「下一關」和「重試」按鈕原本大小和醒目程度相同。實測時我不斷在剛贏的時候誤按「重試」——把勝利的喜悅變成挫折。修正很直接：讓「下一關」在視覺上佔主導，把「重試」降為次要樣式。這是基本的 UX 原則，但只有自己承受過痛苦才會真正內化。

難度曲線無法紙上設計。第一關和第三關都涉及物件持續生成。太快玩家瞬間被淹沒，太慢又覺得無聊。找到那個甜蜜點——「持續緊張但不至於絕望」——需要反覆遊玩，一點一點微調生成速率。再多的理論規劃都無法取代這種手感調校。

視覺重複會扼殺華麗感。第三關的爆炸最初只有一種動畫模式。前幾秒看起來很炫，之後就變成壁紙。加入多種爆炸樣式和粒子變化解決了這個問題——同樣的機制僅僅因為每次爆炸看起來稍有不同，就讓體驗感受截然不同。

技術棧

遊戲是一個單一 HTML 檔案，使用 HTML5 Canvas 做渲染、原生 JavaScript 做遊戲邏輯和狀態管理、CSS3 動畫做 UI 轉場、Touch Events API 做手機觸控操作、多層 Canvas 疊加做視覺層次（煙火特效渲染在獨立圖層上），以及自製的粒子系統處理所有爆炸和慶祝效果。沒有框架、沒有建置工具、沒有相依套件——一個檔案，在瀏覽器打開就能玩。

反思

這個專案最大的收穫不在技術——而在協作模式。透過對話和 AI 一起做遊戲，翻轉了傳統的開發流程。不是先寫程式再測試，而是先玩再描述。最難的部分是把「遊戲手感」——一個本質上模糊的概念——精確到足以讓 Claude 轉化為參數調整。說「感覺太難了」沒有用，說「敵人每 0.8 秒生成一次，我的拇指物理上來不及閃」才有用。

文化元素不只是皮膚。紅包、樂透、爆竹各自天然對應不同的遊戲類型（動作、益智、生存），這意味著主題實際上驅動了玩法多樣性，而不只是裝飾。這是個快樂的發現——最好的設計決策往往來自限制，而非天馬行空的腦力激盪。

如果重來一次，我會加入跨三關的累積計分系統，以及某種社交分享——讓玩家可以傳一張截圖或短動畫給朋友。這遊戲生來就是要被分享的，讓分享變得零摩擦才能完成整個循環。

From Zero to the App Store從零到上架 App Store

By Tim20259 min read閱讀 9 分鐘

Overview

Nihongo Quiz King (日文知識王) is a mobile quiz game where you go head-to-head against an AI opponent, answering Japanese vocabulary questions under time pressure. It started as a personal learning tool — I was studying for the JLPT N4 and wanted something more engaging than flashcards — and ended up becoming my first app published on the App Store. Version 1.0 shipped as a Chinese-only app; version 1.1 added full English localization to reach a wider audience.

Why I Built This

Most Japanese learning apps follow the same pattern: see a word, flip a card, repeat. It works, but it's lonely and low-stakes. I wanted the tension of competition — someone else answering at the same time, a countdown ticking, the pressure of knowing that hesitation costs points. A 1v1 format turns vocabulary review from a chore into a game you actually want to win.

Beyond the product itself, I had a second goal: learn the entire iOS publishing pipeline from scratch. I'm not an iOS developer by background. I wanted to see whether AI-assisted development could take a non-native mobile developer from an empty folder to a live App Store listing — and how painful that journey would actually be.

Core Gameplay

Each match is a rapid-fire series of questions drawn from a pool of 200 JLPT N4 vocabulary words. You see a Japanese word in kana (hiragana/katakana only — no kanji displayed, which matters for Chinese speakers since kanji would essentially give away the answer), and pick the correct meaning from four options within 10 seconds. An AI opponent answers simultaneously, with its accuracy calibrated to your skill level so matches stay competitive.

Speed matters: faster correct answers earn more points. After each question, you see a head-to-head comparison — did you both get it right? Who was faster? At the end of the match, a results screen shows the final score with win/loss animations and sound effects. Your stats persist across sessions: level, cumulative score, and win rate are all tracked locally.

How AI Was Used

Claude was involved in virtually every layer of this project. The collaboration went well beyond "generate some code" — it covered the full stack of shipping a product.

Architecture and code: Claude set up the Expo project structure with expo-router, built all the core game logic (scoring engine, AI opponent behavior, question selection, AsyncStorage persistence), and implemented every screen — home, matchmaking, battle, and results. The animated ring timer (SVG-based countdown) and option button animations (bounce on correct, shake on wrong) were all Claude's work.

Content creation: The entire 200-word N4 question bank was produced through collaboration with Claude — Chinese meanings, English meanings, category tags, and carefully crafted distractor options in both languages. Getting distractors right is harder than it sounds: they need to be plausible enough to challenge the player but not so similar that the question feels unfair.

Asset generation: App icons and splash screens were generated via Python Pillow scripts that Claude wrote. Sound effects (six distinct audio cues for correct, wrong, timeout, victory, defeat, and match start) were also generated programmatically through Python scripts.

Publishing support: Claude helped navigate the entire App Store Connect workflow — writing app descriptions and keywords in both Chinese and English, generating properly sized screenshots, configuring EAS Build settings, and debugging submission issues.

Key Technical Decisions

Expo + EAS over native Xcode. As a non-iOS developer, going fully native would have been a massive detour. Expo with EAS Build let me compile in the cloud without touching Xcode for most of the process. The trade-off is less control over native modules, but for a game that runs entirely in React Native views, it was the right call.

AI opponent instead of real-time multiplayer. Real PvP would require a backend server, matchmaking logic, and infrastructure costs. An AI opponent that adjusts its accuracy based on the player's level gives the same competitive feeling at zero server cost. It's a solo app that feels like a multiplayer one.

Kana-only display, no kanji. This was a game design decision disguised as a UI choice. Japanese kanji and Chinese characters overlap heavily — if the question showed 食べる, any Chinese speaker would immediately know it relates to eating. Showing only たべる forces genuine recall. It's a small detail, but it's the difference between a real quiz and a giveaway.

Auto-detect language, no manual toggle. Version 1.1 added English localization using expo-localization. Rather than adding a language picker to the UI, the app simply follows the device language. One fewer setting to think about, one fewer screen to build, and users get the right language automatically.

The Worst Part: One Week in Review Limbo

Version 1.0 was rejected by App Store review. The reason turned out to be absurdly simple: the name field in app.json didn't exactly match the app name configured in App Store Connect's primary language settings. That's it. A string mismatch.

But finding that answer took a full week. App Store review feedback can be vague, and as a first-time submitter, I didn't have the pattern recognition to quickly narrow down the cause. I cycled through dozens of hypotheses — wrong bundle ID? Missing privacy declarations? Screenshot issues? — before landing on the actual problem.

The fix took 30 seconds. The debugging took seven days. That week cost more calendar time than the entire development phase before it. The lesson is now permanently burned into my process: every time you submit a build, verify that app.json name matches App Store Connect exactly.

Other Painful Lessons

react-native-svg dependency hell. A version conflict between React 19.1 and 19.2 broke the SVG package. The fix required --legacy-peer-deps during local install and adding legacy-peer-deps=true to .npmrc so EAS Build's cloud environment would also cooperate. This kind of issue — works locally, breaks in CI — is uniquely frustrating.

Reanimated v4's naming trap. Version 4 of react-native-reanimated requires a package called react-native-worklets. Version 3 used react-native-worklets-core. Almost the same name, completely different package. Installing the wrong one silently breaks all animations with no useful error message. Hours lost to a hyphen.

App Store Connect's hidden UI. The app name isn't set on the version page — it's under "App Information," a separate section. Adding a new language for localization is buried in a dropdown labeled "Not Yet Localized" on the version page. Every language needs its own screenshots, description, and release notes filled in separately. None of this is obvious to a first-time user.

What Shipped

Version 1.0 launched as a Chinese-only app with the full 200-word question bank, AI opponent, scoring, sound effects, and persistent stats. Version 1.1 added complete English localization — UI strings, question bank (meanings, categories, and distractors all translated), and bilingual App Store metadata. The app is live on the App Store and the source code is open on GitHub.

Reflections

This project proved something I'd suspected but hadn't tested: a non-iOS developer can go from zero to a live App Store product using AI-assisted development, and the result doesn't have to feel like a prototype. The game is genuinely playable, the animations feel polished, and the publishing is real.

The most expensive mistakes weren't technical. They were process mistakes — not understanding App Store conventions, not knowing where settings live in App Store Connect, not anticipating dependency conflicts in cloud build environments. These are the things that no amount of code generation can prevent. You learn them by shipping, failing, and shipping again.

The cross-conversation memory file (MEMORY.md) that Claude and I maintained turned out to be critical. Without it, every new conversation would start from scratch — re-explaining the project structure, the tech stack, the decisions already made. With it, Claude could pick up exactly where we left off. For any multi-session AI project, this is the single highest-leverage practice I'd recommend.

概覽

日文知識王是一款手機答題遊戲，你和 AI 對手面對面，在時間壓力下回答日文單字題目。最初只是一個個人學習工具——我在準備 JLPT N4，想要比單字卡更有趣的東西——最終成為我第一個上架 App Store 的 App。1.0 版以純中文發布；1.1 版加入完整英文本地化，擴大觸及範圍。

為什麼做這個

大多數日文學習 App 都是同一個套路：看單字、翻卡片、重複。有效，但孤單且沒有壓力。我想要競爭的張力——有人同時在回答、倒數在跳動、知道猶豫會扣分。1v1 的模式把單字複習從苦差事變成你真心想贏的遊戲。

除了產品本身，我還有第二個目標：從零學習整個 iOS 上架流程。我不是 iOS 開發者出身。我想看看 AI 輔助開發能否帶一個非原生手機開發者從空資料夾走到上架 App Store——以及這條路到底有多痛苦。

核心玩法

每場對戰是從 200 個 JLPT N4 單字中快速連續出題。畫面顯示假名（平假名/片假名，不顯示漢字——這對中文母語者很重要，因為漢字基本上等於送分），玩家在 10 秒內從四個選項中選出正確意思。AI 對手同時作答，其正確率會根據你的等級調整，讓對戰始終保持競爭性。

速度很重要：答對越快得分越高。每題結束後有即時對比——你們都答對了嗎？誰比較快？對戰結束後的結算畫面顯示最終分數，搭配勝敗動畫和音效。你的數據跨場次保留：等級、累計分數和勝率都在本地追蹤。

AI 如何參與

Claude 參與了這個專案幾乎每一層。協作遠遠超過「幫我生成一些程式碼」——涵蓋了出貨一個產品的完整環節。

架構與程式碼：Claude 建立了 Expo 專案結構（expo-router）、所有核心遊戲邏輯（計分引擎、AI 對手行為、題目選擇、AsyncStorage 持久化），以及每個畫面——首頁、配對頁、對戰頁、結算頁。環形計時器動畫（基於 SVG 的倒數）和選項按鈕動畫（答對彈跳、答錯震動）都是 Claude 的作品。

內容製作：完整的 200 字 N4 題庫都是與 Claude 協作產出——中文字義、英文字義、類別標籤，以及精心設計的中英文干擾選項。做好干擾選項比聽起來更難：它們需要夠有說服力來挑戰玩家，但又不能太相似讓題目感覺不公平。

素材生成：App Icon 和 Splash Screen 透過 Claude 撰寫的 Python Pillow 腳本生成。音效（六種不同的音訊提示：答對、答錯、超時、勝利、失敗、開始）也是用 Python 腳本程式生成的。

上架支援：Claude 協助走完整個 App Store Connect 流程——撰寫中英文的 App 描述和關鍵字、生成正確尺寸的截圖、設定 EAS Build，以及排查送審問題。

關鍵技術決策

選 Expo + EAS，不用原生 Xcode。作為非 iOS 開發者，走全原生會是巨大的繞路。Expo 搭配 EAS Build 讓我可以在雲端編譯，大部分流程不需碰 Xcode。代價是對原生模組的控制力較低，但對一個完全跑在 React Native view 裡的遊戲來說，這是正確的選擇。

AI 對手取代即時多人對戰。真正的 PvP 需要後端伺服器、配對邏輯和基礎建設成本。根據玩家等級調整正確率的 AI 對手，以零伺服器成本給出同樣的競技感。這是一個感覺像多人遊戲的單機 App。

只顯示假名，不顯示漢字。這是一個偽裝成 UI 選擇的遊戲設計決策。日文漢字和中文字高度重疊——如果題目顯示「食べる」，任何中文母語者立刻知道和吃有關。只顯示「たべる」才能強制真正的記憶提取。細節很小，但它是真正考試和直接送分之間的差別。

自動偵測語言，不做手動切換。1.1 版用 expo-localization 加入英文本地化。沒有在 UI 加語言選擇器，App 直接跟裝置語言走。少一個設定要想、少一個畫面要做，用戶自動得到正確的語言。

最痛的部分：退審一週

1.0 版被 App Store 審查退回。原因極其簡單：app.json 的 name 欄位和 App Store Connect 主要語言設定中的 App 名稱不完全一致。就這樣，一個字串不匹配。

但找到這個答案花了整整一週。App Store 審查回饋有時很模糊，作為第一次送審的人，我沒有足夠的經驗來快速縮小原因範圍。我循環測試了幾十個假設——Bundle ID 錯了？隱私聲明漏了？截圖有問題？——最後才找到真正的問題。

修復花了 30 秒。除錯花了七天。那一週花的日曆時間比之前整個開發階段都長。這個教訓現在已經永久刻在我的流程裡：每次送審前，確認 app.json name 和 App Store Connect 完全一致。

其他痛苦的教訓

react-native-svg 依賴地獄。React 19.1 和 19.2 的版本衝突搞壞了 SVG 套件。修復需要在本機安裝時用 --legacy-peer-deps，而且在 .npmrc 加 legacy-peer-deps=true 讓 EAS Build 的雲端環境也能配合。這種「本機正常、CI 爆炸」的問題格外令人抓狂。

Reanimated v4 的命名陷阱。react-native-reanimated v4 需要一個叫 react-native-worklets 的套件。v3 用的是 react-native-worklets-core。名字幾乎一樣，完全不同的套件。裝錯的話所有動畫靜默失效，沒有有用的錯誤訊息。為了一個連字號浪費好幾個小時。

App Store Connect 隱藏的 UI。App 名稱不在版本頁設定——在「App 資訊」，一個獨立的區塊。新增本地化語言藏在版本頁一個寫著「尚未本地化」的下拉選單裡。每個語言都需要各自填寫截圖、描述和更新說明。對第一次使用的人來說完全不直覺。

交付成果

1.0 版以純中文 App 上架，包含完整 200 字題庫、AI 對手、計分、音效和持久化數據。1.1 版加入完整英文本地化——UI 文字、題庫（字義、類別、干擾選項全部翻譯）和雙語 App Store 元資料。App 已在 App Store 上線，原始碼在 GitHub 公開。

反思

這個專案證明了我一直懷疑但未曾驗證的事：一個非 iOS 開發者可以透過 AI 輔助開發從零走到上架 App Store，而且成品不必感覺像是原型。遊戲真正可玩、動畫感覺精緻、上架是真的。

最貴的錯誤不是技術問題。是流程問題——不了解 App Store 慣例、不知道設定藏在 App Store Connect 的哪裡、沒預料到雲端建置環境的依賴衝突。這些是再多程式碼生成都無法預防的。你只能靠出貨、失敗、再出貨來學。

我和 Claude 共同維護的跨對話記憶檔（MEMORY.md）被證明至關重要。沒有它，每次新對話都得從頭解釋——專案結構、技術棧、已經做過的決策。有了它，Claude 可以準確地從上次的地方接手。對任何多次對話的 AI 專案來說，這是我最推薦的單一高槓桿做法。

Cloud Architecture and N3 Expansion雲端架構與 N3 擴充

By Tim20255 min read閱讀 5 分鐘

Overview

With the core game live on the App Store and bilingual support in place, the next challenge was infrastructure. Version 1.2 decoupled the question bank from the app binary, making it updatable without resubmission. Version 1.3 expanded content to JLPT N3 vocabulary and introduced difficulty selection — but doing so exposed a flaw in the original cloud architecture that required a meaningful rethink.

v1.2.0 — Cloud Question Bank

The original app bundled all 200 N4 words directly in the binary. Adding or fixing questions meant rebuilding the app and going through App Store review again — a process that, as documented in the v1.0 article, could take a week for reasons entirely outside your control. The solution was straightforward: host the question bank as a JSON file on GitHub and fetch it at runtime.

But "just fetch from a URL" isn't enough for a mobile app. Network requests fail — planes, subways, spotty connections, GitHub outages. The app needs to work regardless. So v1.2 introduced a three-layer fallback system: try the GitHub-hosted JSON first; if that fails, use the last successfully fetched version cached in AsyncStorage; if there's no cache (first launch, cleared data), fall back to the JSON bundled inside the app. The user always has questions to play with. They just might not be the newest ones.

This took about half a day to implement. The architecture was simple, the behavior was predictable, and it shipped cleanly. What I didn't anticipate was how it would break when the question bank stopped being a single file.

v1.3.0 — N3 Vocabulary and Difficulty Selection

The next step was adding JLPT N3 content — 40 new words at a harder difficulty level — and letting players choose what they want to practice. The UI addition was simple: a difficulty selector before each match offering N4, N3, or a combined pool.

The content work followed the same pattern as v1.0: Claude generated the question entries (Japanese words, meanings in Chinese and English, category tags, and four plausible distractors per language), and I reviewed them for accuracy and pedagogical quality. N3 vocabulary is trickier than N4 — the words are more abstract, the distractors need to be more carefully calibrated, and there are more near-synonyms that could create genuinely ambiguous questions.

The Fallback Architecture Problem

Here's where things got interesting. The v1.2 fallback system treated the question bank as a single monolithic file. When I added N3 as a separate JSON file (n3-words.json alongside n4-words.json), the original logic broke in a subtle way: if the GitHub fetch for either file failed, the entire cloud fetch was considered failed, and the app fell back to the local bundle — which only contained N4 words. Result: N3 mode would show zero questions, with no error message.

The fix was to refactor from a monolithic fallback to a per-source fallback architecture. Each question file now has its own independent fetch-cache-local pipeline:

n4-words.json: GitHub → AsyncStorage cache → local bundled file
n3-words.json: GitHub → AsyncStorage cache → local bundled file
↓
Merge results based on selected difficulty → serve to game

This means a new question file can be pushed to GitHub and immediately available, while a file that hasn't been uploaded yet (or fails to fetch) gracefully falls back to its local version. Each source is independent — one failing doesn't contaminate the other.

It's a small architectural change, but it's the kind of thing that only surfaces when you move from one data source to two. The lesson: fallback logic that works for a single resource often has implicit assumptions about atomicity that break the moment you have multiple resources.

What Shipped

Version 1.3.0 added 40 N3 vocabulary words (bringing the total to 240), a pre-match difficulty selector (N4 / N3 / All), and the per-source fallback architecture. The app is live on the App Store with bilingual support, full sound effects, persistent player stats, and a question bank that can be updated without resubmission.

Reflections

These two versions were smaller in scope than the v1.0→v1.1 journey, but they represent a shift in what the project is about. The first phase was "can I ship an app?" This phase was "can I maintain and grow one?" Different problems: backward compatibility, data architecture, content pipelines, update-without-resubmission strategies.

The per-source fallback refactor is a good example of a pattern I keep seeing in AI-assisted development: the first implementation is fast and clean because the problem is simple. The second iteration reveals assumptions baked into the first design. The value of having Claude as a collaborator isn't just speed — it's being able to refactor quickly when those assumptions surface, rather than living with a brittle design because rewriting feels expensive.

Next up: expanding the N3 bank, potentially adding N5 for beginners, and exploring whether the AI opponent's behavior can be made more nuanced — right now it adjusts accuracy by level, but it could adapt to the player's weak categories specifically.

概覽

核心遊戲已在 App Store 上線、雙語支援也就位後，下一個挑戰是基礎架構。1.2 版把題庫從 App 本體中解耦，讓它可以在不重新送審的情況下更新。1.3 版擴充了 JLPT N3 單字並加入難度選擇——但這麼做暴露了原本雲端架構的一個缺陷，需要真正的重新思考。

v1.2.0 — 雲端題庫

原本的 App 把 200 個 N4 單字直接打包在二進位檔裡。新增或修正題目意味著重新建置 App 並再走一次 App Store 審查——如 v1.0 文章所記錄的，這個過程可能因為完全不受你控制的原因花上一週。解法很直接：把題庫以 JSON 檔案託管在 GitHub，在執行時拉取。

但「從 URL 拉取」對手機 App 來說不夠。網路請求會失敗——飛機上、地鐵裡、訊號不穩、GitHub 故障。App 無論如何都要能運作。所以 1.2 版引入了三層 fallback 系統：先試 GitHub 上的 JSON；失敗就用上次成功拉取的 AsyncStorage 快取；沒有快取（首次啟動、資料被清除）就 fallback 到 App 內建的 JSON。使用者永遠有題目可玩，只是可能不是最新的。

實作花了約半天。架構簡單、行為可預測、乾淨地上線了。我沒預料到的是，當題庫不再是單一檔案時，它會怎麼壞掉。

v1.3.0 — N3 單字與難度選擇

下一步是加入 JLPT N3 內容——40 個更難的新單字——並讓玩家選擇要練什麼。UI 新增很簡單：對戰前的難度選擇器，提供 N4、N3 或全部混合。

內容製作延續 v1.0 的模式：Claude 生成題目條目（日文單字、中英文字義、類別標籤、每種語言四個合理的干擾選項），我審查正確性和教學品質。N3 單字比 N4 棘手——詞彙更抽象、干擾選項需要更仔細校準、有更多近義詞可能造成真正有歧義的題目。

Fallback 架構的問題

有趣的部分來了。v1.2 的 fallback 系統把題庫當成單一整體檔案處理。當我把 N3 加為獨立的 JSON 檔案（n3-words.json 和 n4-words.json 並列），原本的邏輯以微妙的方式壞掉了：如果 GitHub 拉取任一檔案失敗，整個雲端拉取就被視為失敗，App fallback 到本地打包——而本地只有 N4 單字。結果：N3 模式顯示零題，沒有錯誤訊息。

修復是從整體式 fallback 重構為逐來源 fallback 架構。每個題庫檔案現在有自己獨立的拉取-快取-本地管線：

n4-words.json: GitHub → AsyncStorage 快取 → 本地內建檔
n3-words.json: GitHub → AsyncStorage 快取 → 本地內建檔
↓
根據選擇的難度合併結果 → 提供給遊戲

這代表新的題庫檔案推上 GitHub 後立即可用，而尚未上傳（或拉取失敗）的檔案優雅地 fallback 到本地版本。每個來源獨立運作——一個失敗不會污染另一個。

這是一個小的架構改動，但它是那種只有從一個資料源變成兩個時才會浮現的問題。教訓：對單一資源有效的 fallback 邏輯，往往內含關於原子性的隱含假設，在你有多個資源的瞬間就會崩壞。

交付成果

1.3.0 版新增了 40 個 N3 單字（總數達 240）、對戰前難度選擇器（N4 / N3 / 全部），以及逐來源 fallback 架構。App 在 App Store 上線中，具備雙語支援、完整音效、持久化玩家數據，以及無需送審即可更新的題庫。

反思

這兩個版本的規模比 v1.0→v1.1 的旅程小，但它們代表了專案本質的轉變。第一階段是「我能不能出貨一個 App？」這個階段是「我能不能維護和發展一個？」不同的問題：向後相容、資料架構、內容管線、免送審更新策略。

逐來源 fallback 的重構是一個我在 AI 輔助開發中反覆看到的模式的好例子：第一版實作又快又乾淨，因為問題很簡單。第二次迭代揭露了第一版設計中烘焙進去的假設。有 Claude 作為協作者的價值不只是速度——而是當這些假設浮現時能快速重構，而不是因為重寫感覺太貴就忍受脆弱的設計。

接下來：擴充 N3 題庫、可能加入 N5 給初學者，以及探索 AI 對手的行為能否做得更細膩——目前是按等級調整正確率，但可以針對玩家的弱項類別做特定適應。

Memories of Flight飛行的回憶
飛行的回憶

By Tim20259 min read閱讀 9 分鐘
▶ Play in Browser▶ 在瀏覽器遊玩 ⟨/⟩ GitHub

Overview

Memories of Flight (飛行的回憶) is not a game you play to win. It's a game you play to feel something. Built as a single HTML file using Canvas and vanilla JavaScript, it's a narrative experience about loss, regret, and the slow work of reconciliation — told not through cutscenes, but through the act of dodging and catching words as they fall from the sky.

The premise is simple: you control a glowing soul drifting upward through six chapters. White text fragments are memories to catch. Red text fragments are pain to avoid. But the mechanics carry a deeper logic — and the way the screen responds to damage tells you more about the story than any dialogue could.

Why I Built This

Traditional storytelling media — novels, films — let you observe emotion. Interactive media can let you inhabit it. I wanted to test a specific hypothesis: that game mechanics themselves can function as metaphor. When your thumb is physically trembling to dodge a red phrase that reads "it's all your fault," you're not just processing information. You're experiencing a micro-simulation of emotional impact.

The goal wasn't to make something fun. It was to make something that lingers — that players remember not because of a high score, but because of what they felt in the gap between catching a beautiful memory and being immediately punished for it.

Mechanics as Metaphor

The core interaction — catch white words, dodge red words — sounds like any other arcade game. What makes it different is that every design decision maps to an emotional idea.

Collision detection is deliberately uneven. White text (memories) has a generous hit zone. Red text (pain) has a tight one. This isn't a balance problem — it's the point. In life, warmth is hard to hold onto; hurt finds you easily. The asymmetric collision radii encode that feeling directly into the controls.

Catching a memory triggers a danger spike. After you successfully grab a white fragment, the red text spawn rate jumps from 50% to 95%. Every beautiful recollection is immediately followed by a wave of pain. This is the emotional core of the game: remembering something good doesn't bring comfort — it brings everything that came after.

Your soul glows brighter when it remembers. Each white text collected expands the player character's glow radius and quickens its pulse. It's a tiny visual reward — a moment of warmth before the red flood hits.

Two Kinds of Pain

The game's two main chapters use identical core mechanics but completely different damage aesthetics. This was a deliberate design decision rooted in a simple insight: not all pain looks the same.

Chapter 1 — "One Month Ago": The palette is blue-purple. When red text hits you, the screen flashes red and shakes violently. The words shatter into individual character fragments that rain down like debris. This is acute trauma — sharp, loud, impossible to ignore.

Chapter 2 — "Three Months Ago": The palette goes darker, colder. When red text hits, the screen doesn't shake — it slowly loses color, fading toward grayscale. Long sentences crack in half mid-flight and plummet at accelerated speed. This is a different kind of pain: the quiet erosion of a relationship. Words that break off before they finish. The feeling of something draining away rather than exploding.

The technical implementation behind each is distinct: Chapter 1 uses CSS-style screen flash and particle burst effects; Chapter 2 uses getImageData pixel manipulation to progressively desaturate the entire canvas. Same engine, completely different emotional register.

Narrative Structure

The game is organized through a state machine: HOME → STORY → CHAPTER_TITLE → GAMEPLAY → RECAP, cycling through six chapters across two timelines. Between gameplay segments, chapter recaps display three illustrated scenes with typewriter-style text that fades in line by line. At the end of each recap, the scene shatters into particles before transitioning to the next chapter — a visual metaphor for memories fragmenting.

The text content — both the white fragments you catch ("that Friday afternoon," "the text message you sent," "a small shoe rolling across the floor") and the red phrases you dodge ("you only ever loved your work," "I regret everything") — was all written to carry narrative weight. They're not random vocabulary. They're pieces of a story the player assembles through play.

How AI Was Used

Claude's role in this project went beyond code generation into something closer to creative collaboration. When I described abstract emotional goals — "I want the pain in Chapter 2 to feel muffled, not loud" — Claude translated that into a concrete technical proposal: replace the screen shake with progressive desaturation. When I said "the sentence should feel like it was cut short, like someone who stopped talking mid-thought," Claude implemented a text-splitting mechanic triggered at 20% of screen height, with the broken halves accelerating at 2.5x speed.

The codebase still carries traces of our debugging conversations: console logs tagged with emoji (🎯 生成紅字..., 💥 碰撞!..., 📊 統計...) that we used to trace why certain red texts weren't shattering correctly, or why the sentence-split timing was off in Chapter 2. These aren't cleaned up — they're artifacts of a working process.

Claude also handled the deployment pipeline: setting up the GitHub repository, configuring jsdelivr CDN for image assets (https://cdn.jsdelivr.net/gh/{user}/{repo}@main/{path} with crossOrigin = 'anonymous'), and walking through GitHub Pages setup.

Key Design Decisions

Canvas over DOM. Particle systems, real-time pixel manipulation (the grayscale filter), radial glow effects, and screen shake all perform far better on Canvas than with DOM elements. The trade-off is losing built-in accessibility — every UI element had to be drawn manually. For a game that's fundamentally about visual and kinesthetic experience, Canvas was the right medium.

Subtraction as a design principle. Mid-development, the UI had accumulated score displays, timers, hint text, and health indicators. I deliberately removed most of them, keeping only a minimal health bar and a memory score. The reasoning: any element that reminds the player "this is a game" breaks the emotional immersion. In an era where AI can generate infinite content, knowing when to stop adding is the harder skill.

Player size reduction. The player avatar shrank from 50px to 35px during iteration. Smaller meant more precise dodging, which meant each near-miss felt more personal. When you barely escape a red phrase by a few pixels, the relief is visceral in a way a 50px hitbox can't deliver.

Split point at 20%, not 50%. Chapter 2's sentence-breaking mechanic initially triggered at the midpoint of the screen. But at that height, fast-falling text often passed the trigger without being detected. Moving it to 20% gave the split more screen time — players could actually see the sentence crack apart and accelerate — and gave them more (but not enough) reaction time.

Technical Challenges

Sentence split timing. The hardest bug to fix. Text was flying past the split trigger point between frame checks, so sentences that should have broken in half just fell through intact. The solution involved tightening the detection window and adding safety logs to confirm splits were firing. A timing problem disguised as a rendering problem.

Cross-origin image loading. Local image paths broke on GitHub Pages. The fix required hosting images via jsdelivr's GitHub CDN and adding crossOrigin = 'anonymous' to every image load — without it, Canvas getImageData (used for the grayscale effect) throws a security error because the canvas becomes "tainted."

Cross-device scaling. A single scale factor (Math.min(W / 375, H / 667)) applied to all game elements — player size, text size, spawn positions, collision radii — kept the experience consistent across phones and desktops. Both mouse and touch inputs are supported, with touch-action: none preventing accidental zoom on mobile.

What Shipped

A complete, playable narrative game in a single 996-line HTML file. Six chapters across two timelines, each with three illustrated recap scenes. A full visual effects system: particle bursts, screen shake, progressive grayscale, radial glow, static noise, tunnel vision. Cross-device support via dynamic scaling and dual input handling. Deployed on GitHub Pages with CDN-hosted assets. A modular architecture where adding new chapters means extending arrays, not rewriting systems.

Reflections

This project confirmed something I'd believed but hadn't proven: game mechanics can carry emotional weight without a single line of traditional dialogue. When the screen slowly loses color in Chapter 2, players don't need a narrator to explain what's happening — they feel it. The mechanic is the message.

The collaboration model with Claude worked especially well for this kind of project, where the gap between creative intent and technical implementation is wide. I could say "the pain should feel like it's being swallowed, not screamed" and get back a concrete implementation using pixel-level desaturation instead of screen flash. That translation layer — from emotion to code — is where AI collaboration adds the most value in creative work.

If I were to build on this, I'd add an ambient soundtrack that shifts with the chapter mood, and explore whether the finale could use the accumulated memories (the white texts the player caught) to generate a personalized ending. The architecture supports it. The emotional logic demands it.

概覽

飛行的回憶不是一款讓你玩來贏的遊戲。它是一款讓你玩來感受的遊戲。以單一 HTML 檔案、Canvas 和原生 JavaScript 打造，它是一段關於失去、悔恨與和解的敘事體驗——不是透過過場動畫來訴說，而是透過閃避和接住從天而降的文字這個動作本身。

前提很簡單：你操控一個發光的靈魂，緩緩穿越六個章節。白色文字碎片是要接住的回憶。紅色文字碎片是要閃避的傷痛。但機制承載著更深的邏輯——畫面對傷害的反應方式，比任何對白都更能說明這個故事。

為什麼做這個

傳統敘事媒介——小說、電影——讓你觀察情緒。互動媒介可以讓你住進情緒裡。我想驗證一個具體的假設：遊戲機制本身可以作為隱喻。當你的拇指物理上在顫抖著閃避一段寫著「都是你的錯」的紅色文字時，你不只是在處理資訊，你在經歷一次微型的情緒衝擊模擬。

目標不是做一個好玩的東西，而是做一個會留下來的東西——玩家記住它不是因為高分，而是因為在接住一段美好回憶後立刻被懲罰的那個間隙裡，他們感受到的。

機制即隱喻

核心互動——接住白字、閃避紅字——聽起來和任何街機遊戲一樣。讓它不同的是，每個設計決策都對應一個情緒概念。

碰撞判定刻意不對稱。白色文字（回憶）有寬鬆的判定範圍。紅色文字（傷痛）有嚴格的判定範圍。這不是平衡問題——這就是重點。在人生中，溫暖難以抓住；傷害輕易找上你。不對稱的碰撞半徑把這種感受直接編碼進操控裡。

接住回憶會觸發危險激增。成功抓到白色碎片後，紅字的生成率從 50% 飆升到 95%。每一段美好的回憶都立刻接著一波痛苦。這是遊戲的情緒核心：想起美好的事不會帶來安慰——它帶來的是之後發生的一切。

你的靈魂在記住時會發光更亮。每接住一段白色文字，玩家角色的發光範圍會擴大、脈動加快。這是一個微小的視覺獎勵——紅色洪流來臨前的一刻溫暖。

兩種不同的痛

遊戲的兩個主要章節使用相同的核心機制，但完全不同的傷害美學。這是一個刻意的設計決策，根植於一個簡單的洞察：不是所有的痛看起來都一樣。

第一章——「一個月前」：色調偏藍紫。當紅字擊中你時，畫面紅光閃爍並劇烈震動。文字碎裂成單字碎片如雨般落下。這是急性創傷——尖銳、響亮、無法忽視。

第二章——「三個月前」：色調更暗、更冷。紅字擊中時畫面不會震動——而是慢慢失去色彩，逐漸褪向黑白。長句子在飛行途中斷裂成兩半，加速墜落。這是不同的痛：一段關係的靜默侵蝕。話說到一半就斷了。某種東西在流逝而非爆炸的感覺。

背後的技術實作各有不同：第一章用 CSS 風格的螢幕閃光和粒子爆發效果；第二章用 getImageData 像素操作，逐步去飽和整個畫布。同一個引擎，完全不同的情緒調性。

敘事結構

遊戲透過狀態機組織：HOME → STORY → CHAPTER_TITLE → GAMEPLAY → RECAP，在兩條時間線的六個章節中循環。在遊戲環節之間，章節回顧顯示三幕插畫場景，搭配逐行淡入的打字機風格文字。每段回顧結束時，場景碎裂成粒子後轉場到下一章——一個記憶碎裂的視覺隱喻。

文字內容——無論是你接住的白色碎片（「那個週五下午」、「妳傳來簡訊」、「滾落的小球鞋」）還是閃避的紅色語句（「你只愛工作」、「我好後悔」）——都是帶著敘事份量寫的。它們不是隨機詞彙，而是玩家透過遊玩拼湊出的故事碎片。

AI 如何參與

Claude 在這個專案中的角色超越了程式碼生成，更接近創意協作。當我描述抽象的情緒目標——「我希望第二章的痛感覺是悶住的，不是大聲的」——Claude 把它翻譯成具體的技術方案：用漸進式去飽和取代螢幕震動。當我說「句子應該感覺像是被截斷了，像某個人說到一半就不說了」，Claude 實作了一個在螢幕高度 20% 處觸發的文字斷裂機制，斷裂的兩半以 2.5 倍速加速。

程式碼裡至今留著我們除錯對話的痕跡：帶 emoji 標記的 console log（🎯 生成紅字...、💥 碰撞!...、📊 統計...），我們用來追蹤為什麼某些紅字沒有正確碎裂，或者第二章的句子斷裂時機為什麼不對。這些沒被清理——它們是工作過程的遺跡。

Claude 也負責了部署流程：建立 GitHub 倉庫、設定 jsdelivr CDN 託管圖片資源（https://cdn.jsdelivr.net/gh/{user}/{repo}@main/{path} 搭配 crossOrigin = 'anonymous'），以及引導 GitHub Pages 設定。

關鍵設計決策

Canvas 勝過 DOM。粒子系統、即時像素操作（灰階濾鏡）、徑向發光效果和螢幕震動，在 Canvas 上的表現遠優於 DOM 元素。代價是失去內建的無障礙性——每個 UI 元素都需要手動繪製。對於一款本質上關乎視覺與動覺體驗的遊戲，Canvas 是正確的媒介。

減法作為設計原則。開發中期，UI 累積了分數顯示、計時器、提示文字和血量指示器。我刻意移除了大部分，只留最精簡的血條和記憶分數。理由：任何讓玩家「想起這是個遊戲」的元素，都會打斷情緒沉浸。在 AI 能無限產出內容的時代，知道何時停止添加才是更難的技藝。

縮小玩家尺寸。玩家角色在迭代中從 50px 縮小到 35px。更小意味著更精準的閃避，這讓每次險險避開都更加私人化。當你僅以幾個像素之差逃過一段紅色語句時，那種如釋重負是 50px 判定框無法給予的。

斷裂點在 20%，不是 50%。第二章的句子斷裂機制最初在螢幕中點觸發。但在那個高度，快速下落的文字經常在幀檢查之間越過觸發點。移到 20% 讓斷裂有更多螢幕時間——玩家真的可以看到句子裂開並加速——也給了他們更多（但仍不夠）的反應時間。

技術挑戰

句子斷裂的時序。最難修的 bug。文字在幀檢查之間飛過斷裂觸發點，本該斷成兩半的句子原封不動地飛過去了。解法是收緊偵測窗口，並加上安全日誌確認斷裂有在觸發。一個偽裝成渲染問題的時序問題。

跨域圖片載入。本地圖片路徑在 GitHub Pages 上失效。修復需要透過 jsdelivr 的 GitHub CDN 託管圖片，並在每次圖片載入時加上 crossOrigin = 'anonymous'——否則 Canvas 的 getImageData（用於灰階效果）會因為畫布被「污染」而拋出安全錯誤。

跨裝置縮放。一個單一縮放因子（Math.min(W / 375, H / 667)）套用到所有遊戲元素——玩家大小、文字大小、生成位置、碰撞半徑——讓手機和桌機的體驗保持一致。同時支援滑鼠和觸控輸入，以 touch-action: none 防止手機上的意外縮放。

交付成果

一個完整可玩的敘事遊戲，濃縮在 996 行的單一 HTML 檔案裡。兩條時間線的六個章節，每章三幕插畫回顧場景。完整的視覺特效系統：粒子爆發、螢幕震動、漸進式灰階、徑向發光、靜態雜訊、隧道視野。透過動態縮放和雙輸入處理支援跨裝置。部署在 GitHub Pages，圖片資源由 CDN 託管。模組化架構——新增章節只需擴充陣列，不需重寫系統。

反思

這個專案確認了我一直相信但未曾證明的事：遊戲機制可以承載情緒重量，不需要一行傳統對白。當畫面在第二章中慢慢失去色彩時，玩家不需要旁白解釋正在發生什麼——他們感受得到。機制就是訊息。

和 Claude 的協作模式在這類專案中特別有效，因為創意意圖和技術實作之間的落差很大。我可以說「痛應該感覺像是被吞下去的，不是被吼出來的」，然後得到一個用像素級去飽和取代螢幕閃光的具體實作。那個翻譯層——從情緒到程式碼——是 AI 協作在創意工作中最能增值的地方。

如果要繼續發展，我會加入隨章節情緒變化的環境音效，並探索結局能否使用累積的記憶（玩家接住的白色文字）生成個人化的結尾。架構支援這麼做。情緒邏輯也要求這麼做。

Bilingual-izing This Site with AI用 AI 把這個網站雙語化

By Tim20255 min read閱讀 5 分鐘

Overview

This article is about itself — or rather, about the process of making it readable in two languages. This portfolio site started as an English-only, 1,200-line static HTML file. I wanted to add full Traditional Chinese support with a one-click toggle, and I used Claude to do it. The process was instructive not because it was hard, but because of how it failed before it succeeded.

The Goal

Add bilingual EN/ZH support to every piece of text on the site: navigation, hero section, project cards, about section, footer, and four full-length project articles. The user clicks a button in the top-right corner, and everything switches instantly. Their preference persists across page reloads via localStorage.

The Approach That Didn't Work

The first attempt was straightforward: rewrite the entire file with bilingual markup in one shot. Claude would take the 1,200-line HTML file, wrap every text element with <span class="lang-en"> and <span class="lang-zh"> tags, add the CSS toggle rules, insert the JavaScript, and output a complete new file.

It failed. Repeatedly. The file was too large for a single coherent rewrite — Claude would lose track of the structure partway through, producing truncated or malformed output. This happened multiple times before I (the human) called it out: "you keep failing, you need to change your approach."

In hindsight, the failure makes sense. A 1,200-line file with deeply nested HTML, long-form article content in English that needs Chinese equivalents, plus new CSS and JavaScript — that's a lot of moving parts to hold in context simultaneously. The task isn't intellectually hard, but it's volumetrically hard. There's a difference.

The Approach That Worked

The fix was to decompose the work into sequential, verifiable steps:

Step 1 — Infrastructure: Add the CSS language toggle rules ([lang="en"] .lang-zh { display: none } and vice versa), the nav toggle button, and the JavaScript switching function. Don't touch any content yet. Verify the button appears and the CSS logic works.

Step 2 — Homepage: Translate the hero, project cards, about section, and footer. Checkpoint: download, preview, confirm the toggle switches all homepage text correctly.

Step 3 — Articles, one at a time: Wrap each article's English content in <span class="lang-en">, write the full Chinese translation in <span class="lang-zh">, and add it to the file. Checkpoint after each article.

Each step produced a working, downloadable file. If something broke, the blast radius was one article, not the entire site. The checkpoints weren't just nice-to-have — they were the mechanism that made the whole process reliable.

Technical Implementation

The i18n system is deliberately minimal. No framework, no JSON translation files, no build step. Just CSS and HTML:

The <html lang="en"> attribute controls which language is visible. CSS rules hide the non-active language: [lang="en"] .lang-zh { display: none !important }. A JavaScript function toggles the lang attribute and saves the preference to localStorage. On page load, it checks localStorage and applies the saved language.

Every translatable element contains both versions inline: <span class="lang-en">Projects</span><span class="lang-zh">專案</span>. For long article content, the entire body is wrapped in two sibling spans — one per language. This means the HTML file is roughly twice its original size, but for a static site with no build process, the simplicity is worth the payload.

What I Learned

Volume is a different kind of complexity. Bilingual-izing a website isn't conceptually difficult. It's wrapping text in spans and writing translations. But when the file is 1,200+ lines with four long-form articles, the sheer volume overwhelms a single-pass approach. The lesson applies beyond i18n: any large-scale transformation of existing content is better done incrementally than all at once, especially with AI.

Checkpoints are infrastructure, not overhead. Every step ended with "download this, verify it works." That discipline caught issues early and kept the working version intact. Without it, a single bad edit deep in the file could have corrupted hours of work with no easy rollback.

The human's job is strategy, not content. I didn't write any of the Chinese translations — Claude did all of that. What I contributed was the decision to stop the failing approach, the decomposition into steps, and the verification at each checkpoint. In AI collaboration, knowing when to change the plan is more valuable than knowing how to execute the plan.

Simple beats clever for static sites. There are elegant i18n solutions — JSON translation files, framework-level routing, compile-time generation. For a single-file static site on GitHub Pages, inline spans with CSS toggles are the right level of complexity. It's not scalable to 50 languages, but it doesn't need to be. It needs to work for two, today, with zero dependencies.

The Meta Observation

This article exists because the process of building it was interesting enough to document. The portfolio site was built by AI, the articles were written by AI, and the bilingual system was implemented by AI. But the decisions — what to build, when to change strategy, what quality bar to set — were all human. That division of labor is, in miniature, the thesis of this entire portfolio.

概覽

這篇文章講的是它自己——或者更精確地說，是讓它能被兩種語言閱讀的過程。這個作品集網站最初是一個純英文、1,200 行的靜態 HTML 檔案。我想加入完整的正體中文支援，按一個鍵就能切換，並用 Claude 來實作。這個過程很有啟發性，不是因為它很難，而是因為它怎麼失敗然後才成功的。

目標

為網站上的每一段文字加入雙語 EN/ZH 支援：導航列、首頁大標、專案卡片、關於區塊、頁尾，以及四篇完整的專案文章。使用者點一下右上角的按鈕，所有內容立即切換。偏好設定透過 localStorage 在重新載入頁面後保留。

行不通的做法

第一個嘗試很直接：一次重寫整個檔案，加入雙語標記。Claude 會拿到 1,200 行的 HTML 檔案，在每個文字元素外包上 <span class="lang-en"> 和 <span class="lang-zh"> 標籤，加入 CSS 切換規則，插入 JavaScript，然後輸出一個完整的新檔案。

失敗了。而且重複失敗。檔案太大，無法在一次通過中保持連貫——Claude 寫到一半就會失去對結構的掌控，產出截斷或格式錯誤的輸出。這發生了好幾次，直到我（人類）指出：「你一直失敗，你得換個做法。」

事後看來，失敗是合理的。一個 1,200 行、深度巢狀的 HTML 檔案，加上需要中文對應的長篇英文文章內容，再加上新的 CSS 和 JavaScript——這是同時要在上下文中維持的大量移動部件。這個任務在智識上不難，但在體量上很難。兩者是不同的。

行得通的做法

修正方法是把工作分解為循序的、可驗證的步驟：

步驟一——基礎架構：加入 CSS 語言切換規則（[lang="en"] .lang-zh { display: none } 及反向規則）、導航列的切換按鈕，以及 JavaScript 切換函式。不碰任何內容。驗證按鈕出現且 CSS 邏輯運作。

步驟二——首頁：翻譯 Hero 區塊、專案卡片、關於區塊和頁尾。存檔點：下載、預覽、確認切換能正確切換所有首頁文字。

步驟三——文章逐篇處理：把每篇文章的英文內容包在 <span class="lang-en"> 裡，寫完整的中文翻譯放在 <span class="lang-zh"> 裡，加入檔案。每篇文章後設存檔點。

每一步都產出一個可運作、可下載的檔案。如果出問題，爆炸半徑是一篇文章，不是整個網站。存檔點不只是「有也不錯」——它們是讓整個過程可靠的機制。

技術實作

i18n 系統刻意保持精簡。沒有框架、沒有 JSON 翻譯檔、沒有建置步驟。只有 CSS 和 HTML：

<html lang="en"> 屬性控制哪個語言可見。CSS 規則隱藏非啟用的語言：[lang="en"] .lang-zh { display: none !important }。一個 JavaScript 函式切換 lang 屬性並把偏好存到 localStorage。頁面載入時檢查 localStorage 並套用儲存的語言。

每個可翻譯的元素都包含兩個版本：<span class="lang-en">Projects</span><span class="lang-zh">專案</span>。對於長篇文章內容，整個 body 包在兩個並列的 span 裡——每個語言一個。這意味著 HTML 檔案大約是原來的兩倍大，但對於一個沒有建置流程的靜態網站來說，簡單性值得這個代價。

我學到的

體量是一種不同的複雜性。把網站雙語化在概念上不難。就是在文字外面包 span 然後寫翻譯。但當檔案超過 1,200 行、有四篇長文章時，純粹的體量就會壓垮一次通過的做法。這個教訓不只適用於 i18n：任何對現有內容的大規模轉換，用漸進式做比一次全改更好，特別是和 AI 協作時。

存檔點是基礎設施，不是額外負擔。每一步都以「下載這個，驗證它能動」結束。這個紀律在早期就能抓到問題，並保持工作版本完整。沒有它，一個藏在檔案深處的錯誤編輯可能毀掉數小時的工作，而且沒有簡單的回退方式。

人類的工作是策略，不是內容。我沒有寫任何一行中文翻譯——全部是 Claude 做的。我貢獻的是決定停止失敗的做法、把工作分解成步驟、以及在每個存檔點進行驗證。在 AI 協作中，知道何時該改變計劃比知道如何執行計劃更有價值。

對靜態網站來說，簡單勝過精巧。有很多優雅的 i18n 方案——JSON 翻譯檔、框架層級的路由、編譯時生成。對於一個放在 GitHub Pages 上的單檔靜態網站，用 inline span 加 CSS 切換是正確的複雜度層級。它無法擴展到 50 種語言，但它也不需要。它只需要今天能為兩種語言工作，零相依套件。

Meta 觀察

這篇文章之所以存在，是因為建造它的過程本身有趣到值得記錄。作品集網站是 AI 建的，文章是 AI 寫的，雙語系統是 AI 實作的。但決策——做什麼、何時改變策略、設定什麼品質標準——全部是人類做的。這種分工，就是整個作品集的論點的縮影。

Project 06 — Automation × AI

The Job Monitor That Lied for 27 Days那個騙了我 27 天的求職監控系統

By Tim20259 min read閱讀 9 分鐘

Overview

I was running a multi-market, multi-platform job search and getting buried in noise. Every job site's built-in "save this search, email me matches" feature has a terrible signal-to-noise ratio — you end up scanning dozens of irrelevant postings to find the one that matters. So I built a small system to do it for me: scrape a few platforms daily, score every posting against what I'm actually looking for, and email me only the ones that clear a threshold. It runs on GitHub Actions, costs nothing, and notifies me by email every morning. It also, for 27 days, ran perfectly and found nothing — without telling me that anything was wrong. This is the story of that bug.

The One Rule: Find and Notify, Never Auto-Apply

The original idea was bigger: auto-find and auto-apply. I dropped the second half almost immediately, for three reasons. A resume that isn't tailored to the specific posting dilutes your signal — sending the same one to fifty jobs is worse than sending five tailored ones. Job platforms detect automated behavior and can restrict the account you actually need for your real search. And a posting whose title looks like a match is, surprisingly often, a different kind of role entirely once you read the description — auto-applying on title alone wastes real opportunities. So the system only ever does two things: find, and notify. A human decides what to do next.

The Architecture (and a Decision That Would Come Back to Haunt Me)

The system is unglamorous on purpose: a GitHub Actions cron job runs daily, a scraper searches a handful of keywords across a couple of platforms, a scoring function filters the results, and a notify script emails the survivors over Gmail SMTP. Already-seen postings are tracked in a JSON file committed to the repo — no database needed.

The first real decision was how to fetch the data. Calling a platform's internal search API directly with plain HTTP requests returned 403 every time — it wanted cookies, a referer, full browser-shaped headers. The fix was to stop pretending to be an API client and instead drive an actual headless Chromium browser with Playwright, load the real search page, and parse the rendered DOM. From the platform's point of view, that's indistinguishable from a person browsing. Anti-bot defenses aimed at API clients have nothing to say about a real browser.

It was the right call at the time. It also turned out, months later, to be the exact decision that broke everything — for reasons that had nothing to do with bots.

From Blacklist to a Scoring System

The first filtering approach was a blacklist: exclude postings whose title contained certain words, exclude postings from certain companies. It fell apart within days. The exclusion list never stopped growing — every new flavor of noise needed its own new rule. It misfired constantly — blocking one ambiguous word in a title would also block a posting that was actually a perfect match. And it was unexplainable — there was no way to answer "why did this get filtered out?"

The fix was to stop thinking in yes/no and start scoring every posting on a point scale, with a threshold for notification. A title that exactly matches the target role scores high enough to clear the bar by itself. A title with a related-but-not-exact term scores a smaller bonus. A title with an obviously different role scores a heavy penalty. Ambiguous seniority words in a title get only a token penalty — light enough that a strong description can still pull the score back up. Description keywords add a little, capped so keyword-stuffing can't game it. A company name that's a strong negative signal subtracts enough to override a good title score on its own.

The bigger design choice underneath all of this was choosing loose over strict. A strict system blocks anything ambiguous and produces less noise, but quietly drops the postings whose titles are understated but whose descriptions are exactly right. A loose system lets ambiguous titles through and lets the description redeem them, producing more noise but losing fewer real opportunities. I chose loose, because in a job search, missing a good opportunity costs far more than spending an extra minute skimming past a bad one.

The Bug That Wore a Green Checkmark for 27 Days

The system shipped, ran clean for about a week, and I stopped watching the logs — which is exactly when it broke. About three weeks later I noticed I hadn't gotten an email in a while. I went to check the Actions history: every single run, for roughly a month, had a green checkmark. And every single run had found zero new postings. No errors. No failed steps. Just twenty-some consecutive clean runs that had quietly done nothing.

A green checkmark turned out to be the most dangerous failure mode there is — it doesn't just fail to alert you, it actively tells you everything is fine.

Fixing it took three rounds, and the first two both looked, at the time, like the real answer.

Act One — the wrong mountaintop: Running the scraper locally with step-by-step logging showed every search hanging at the page-load step, then timing out after 30 seconds. The wait condition was networkidle — wait until the network goes quiet for 500ms before continuing. But the target page kept a steady stream of third-party tracking scripts firing in the background, so the network never went quiet, and the wait never resolved. The timeout got caught by a try/except and the code moved on to the next keyword — which is why the whole workflow reported success. The fix looked obvious: switch to domcontentloaded plus a more specific wait_for_selector for the job card element, so the code stops waiting for the entire network and instead waits for the one thing it actually needs. Locally, postings started coming through again. It looked solved.

Act Two — the selectors were stale too: The data that came back was messy. Dumping the rendered HTML and reading it line by line showed the platform had also changed its DOM structure since the scraper was written. New selectors for the date, location, and description fields fixed most of it. The title field was the interesting one: the old code read textContent off the link, but the new markup wrapped the user's search term in a highlight <span> inside the title, so the extracted text came out with stray fragments mixed in. Reading the title attribute instead gave back the clean string. Local runs were now stable, pulling in a normal day's worth of postings. I committed it and pushed.

Act Three — local was fine, the cloud was still zero: The next morning's Actions run came back with zero postings. Same code, same commit — twenty-some postings locally, zero on GitHub's runner. That's the moment it became clear the first two fixes had only ever been treating symptoms. The variable that mattered wasn't timing and it wasn't selectors. It was the environment itself.

To actually measure that, I wrote a small diagnostic script that used nothing but the standard library — no browser at all — wired it into a throwaway, manually-triggered workflow, and let it run directly on a GitHub Actions runner. It checked three things at once: the runner's outbound IP and rough location, a direct HTTP call to the platform's underlying JSON API, and the raw HTML of the search page itself.

The results were unambiguous. The runner sat on a US datacenter IP. The raw HTML of the search page was a near-empty shell — a few kilobytes, zero job links in it. And the direct HTTP call to the platform's own API came back with a full 200 and a complete page of results.

The platform, it turned out, had quietly become a pure front-end single-page app at some point. The server now sends an empty shell; the actual postings are fetched by a client-side XHR call to an API and rendered in the browser afterward. Locally, Playwright's full Chromium executes that JavaScript, the XHR goes through, and postings render normally. On the runner, the exact same XHR call — made by a headless browser sitting on a datacenter IP — was getting silently blocked by anti-bot defenses. But a plain HTTP request to that same API, from that same datacenter IP, sailed through with no problem at all.

That distinction mattered enormously: the failure was about rendering, not about IP or geography. Mistake that for a geo-block and you'd reach for a proxy or a different region of VPS — entirely the wrong fix.

The actual fix — remove the browser: Since the API itself answered fine from a datacenter IP, the cleanest solution was to stop driving a browser entirely. The scraper now makes a plain HTTP request straight to the platform's JSON API and maps the response fields onto the same internal job format the rest of the pipeline already expected. The side benefits stacked up immediately: no more installing Chromium in CI, no headless-browser fingerprint to get flagged in the first place, and salary and date fields that came out of the API cleaner than anything scraped from the DOM ever did.

There's a particular irony here. This fix walks straight back into the path that the original architecture decision explicitly rejected — calling the platform's API directly. The "403 for everyone" conclusion from months earlier turned out to no longer hold; a plain user agent and referer, no cookies required, was enough. The decision that once felt like the clever way around the platform's defenses had become, by the time this bug surfaced, the actual cause of the outage — and the thing it had been built to avoid was the fix that replaced it.

What Actually Saved the Debug

This bug got handed off mid-investigation, in a different mode of working with Claude — pasting logs back and forth in a chat window is slow; every "try this" means switching to a terminal, copying output, pasting it back, at three to five minutes per round trip, with no visibility into the actual local environment. Working directly inside the project, where Claude could run the scraper itself, read its own output, and edit files immediately, cut that loop to seconds. But speed wasn't actually the thing that mattered here.

Two things were. The first: being able to measure inside the actual broken environment, not just a proxy for it. Claude Code itself runs locally — it isn't the GitHub Actions runner either — but it could write a diagnostic script, wire it into a disposable workflow, and have that workflow execute on the real runner, then read the results back. That's the step that actually separated "local works" from "cloud doesn't" instead of just guessing at the gap between them.

The second, and more important one: after Act Two, the fix looked finished, and it would have been easy to call it done right there. What stopped that was a single skeptical question — "are you sure the API actually works from the cloud, or could that be different from local too?" That question is what triggered measuring the API on the real runner before committing to a rewrite. Without it, this would very plausibly have shipped as another "fixed it" that quietly stayed broken. Good collaboration with AI isn't only about how fast it can execute — it's also about a human applying the brakes at exactly the right moment.

Lessons

A green checkmark is not health. What matters is the actual output count, and "zero results for N consecutive days" deserves its own active alert — silent failure is the most dangerous failure.

Don't diagnose only on the machine that works. Local (residential IP, full browser) and cloud (datacenter IP, headless) behave differently in ways that matter. A fix that only gets verified locally hasn't actually been verified where it counts.

Separate the root-cause category before reaching for a fix. "Rendering is blocked" and "the IP is blocked" look identical from the outside and require completely different solutions. A few minutes spent building a probe to tell them apart is cheaper than guessing wrong and rewriting twice.

When a page is a SPA, look for the API underneath it first. Calling the API directly is almost always faster, more stable, and harder for anti-bot systems to flag than driving a full browser to render the same data.

The decision you're proudest of can be the one you later have to undo. The headless-browser workaround was the right call against the problem it was solving. It just wasn't the right call forever.

Final Numbers

The system now runs every morning, unattended. A typical run pulls in roughly 200 raw postings; after scoring, usually 30 to 60 clear the notification threshold. Removing the browser also cut the per-run time from a few minutes (most of it installing Chromium) down to a few seconds. GitHub Actions usage stays well inside the free tier, and Gmail SMTP is free. Total monthly cost: zero.

概覽

我同時在多個市場、多個平台上找工作，很快就被雜訊淹沒。幾乎每個求職平台內建的「儲存搜尋條件、用 Email 通知我」功能，訊噪比都很差——你得掃過幾十筆不相關的職缺，才找到一筆真正重要的。所以我做了一個小系統幫我做這件事：每天自動掃幾個平台、把每筆職缺拿去跟我真正在找的東西打分數、只有過門檻的才用 Email 通知我。它跑在 GitHub Actions 上，零成本，每天早上準時寄信。它也曾經連續 27 天「完美運作」、什麼都沒找到——而且完全沒告訴我哪裡出了問題。這是那個 bug 的故事。

唯一的原則：只找、只通知，永遠不自動投

最初的想法更大：自動找、自動投。我幾乎立刻就放棄了後半段，理由有三個。沒有針對職缺客製化的履歷會稀釋你的訊號——同一份履歷投五十個職缺，遠比不上五份客製化履歷投五個。求職平台會偵測自動化行為，進而限制你真正需要的那個帳號。而且標題看起來像目標職務的職缺，打開描述一看常常根本是完全不同性質的工作——光靠標題自動投，浪費的是真正的機會。所以這個系統永遠只做兩件事：找，和通知。下一步永遠由人決定。

架構（以及一個後來反咬自己一口的決定）

系統本身刻意做得很樸實：一個 GitHub Actions 排程每天跑一次，scraper 用幾個關鍵字搜尋幾個平台，評分函式過濾結果，notify 腳本透過 Gmail SMTP 把過關的職缺寄出去。已經通知過的職缺記錄在一個 commit 進 repo 的 JSON 檔裡——不需要資料庫。

第一個真正的決定，是怎麼抓資料。直接用 Python 的 HTTP request 打平台的內部搜尋 API，每次都拿到 403——平台需要 cookie、Referer、完整的瀏覽器標頭才肯回應。修法是不再假裝自己是個 API client，改用 Playwright 開一個真正的 headless Chromium，載入真實的搜尋頁面，解析渲染後的 DOM。從平台的角度看，這跟真人開瀏覽器逛網站沒有差別。針對 API client 設計的反爬蟲機制，對一個真實瀏覽器毫無辦法。

在當下，這是正確的決定。但幾個月後事實證明，正是這個決定，後來把整個系統搞壞——而原因跟「機器人偵測」完全無關。

從黑名單到評分制

第一版的過濾方式是黑名單：標題含特定字就排除，公司是特定名單上的就排除。跑了幾天就垮了。排除清單永遠列不完——每出現一種新雜訊，就要再加一條新規則。它常常誤殺——擋掉標題裡一個模糊的字，連帶擋掉了一筆其實完全符合的職缺。而且它無法解釋——沒辦法回答「這筆職缺為什麼被擋掉？」

修法是不再用是非題思考，改成幫每筆職缺打分數，達到門檻才通知。標題完全符合目標職務的，單靠這一條就能過門檻。標題相關但不完全一致的詞，給比較小的加分。標題明顯是別種職務的，扣很重的分。標題裡模糊的職等字眼，只給象徵性的小扣分——輕到讓一段夠強的描述還能把分數拉回來。描述裡的關鍵字小幅加分，且設有總上限，避免靠堆關鍵字灌分。一個本身就是強烈負面訊號的公司名稱，扣分要重到足以單獨蓋過一個不錯的標題分數。

底下更大的設計取捨，是選擇寬鬆而不是嚴格。嚴格的系統會擋掉任何模糊的東西、雜訊較少，但會悄悄漏掉那些標題含蓄、描述卻完全符合的好職缺。寬鬆的系統讓模糊標題先過關，靠描述去把它救回來或刷掉，雜訊較多，但漏掉的真機會更少。我選了寬鬆，因為在求職這件事上，漏掉一個好機會的成本，遠高於多花一分鐘掃過一筆雜訊。

那個戴著綠勾勾撒了 27 天謊的 bug

系統上線、穩定跑了大約一週後，我就不再盯著 log 看了——而這正是它開始壞掉的時刻。大約三週後，我注意到好一段時間沒收到信了。回去查 Actions 紀錄：將近一個月，每一次執行都是綠勾勾。而每一次執行，抓到的新職缺都是零筆。沒有任何錯誤訊息，沒有任何失敗的步驟。就只是二十幾次乾乾淨淨、什麼事都沒做的「成功」執行。

綠勾勾原來是最危險的故障模式——它不只是沒有提醒你，它還主動告訴你一切正常。

修好它花了三個回合，前兩個回合在當下都看起來像是真正的答案。

第一幕——錯誤的山頂：在本機直接跑 scraper、加上逐步 log，看到每個搜尋都卡在頁面載入那一步，然後 30 秒後 timeout。等待條件是 networkidle——要等網路靜止 500ms 才繼續往下走。但目標頁面背景持續發送第三方追蹤腳本的請求，網路永遠不會靜下來，等待條件永遠不會被滿足。Timeout 被 try/except 接住，程式接著處理下一個關鍵字——這就是為什麼整個 workflow 顯示「成功」。修法看起來很明顯：換成 domcontentloaded，再加一個更明確的 wait_for_selector 等職缺卡片元素出現，讓程式不再等整個網路安靜，而是只等它真正需要的那個東西。本機跑起來，職缺又開始抓到了。看起來解決了。

第二幕——selector 也過期了：抓回來的資料卻是髒的。把渲染後的 HTML dump 出來逐行檢查，發現平台從寫這支 scraper 之後也改了整個 DOM 結構。換上日期、地點、描述欄位新的 selector，解決了大部分問題。標題欄位最有意思：舊程式碼讀的是連結的 textContent，但新版的標記裡，使用者搜尋的關鍵字被包在標題裡一個用來高亮的 <span> 中，抓出來的文字夾雜著奇怪的片段。改成讀 title 屬性，拿到的就是乾淨的字串。本機這下穩定下來，一次抓到一天該有的職缺量。我把它 commit、push 上去。

第三幕——本機沒事，雲端還是零：第二天早上的 Actions 執行結果回來，還是零筆。同一份程式碼、同一個 commit——本機二十幾筆，GitHub 的 runner 上零筆。就是這一刻，前兩次修法的本質才浮現：它們從頭到尾都只是治標。真正關鍵的變因不是等待時機，也不是 selector，而是環境本身。

為了真的把這件事量出來，我寫了一支只用標準函式庫、完全不開瀏覽器的診斷腳本，接進一支拋棄式、手動觸發的 workflow，讓它直接跑在 GitHub Actions 的 runner 上。它同時量三件事：runner 對外的 IP 和大致地理位置、直接用 HTTP 打平台底層的 JSON API、以及搜尋頁本身的原始 HTML。

結果一翻兩瞪眼。runner 的 IP 是美國機房。搜尋頁的原始 HTML 幾乎是個空殼——只有幾 KB，裡面零個職缺連結。而直接打平台自己的 API，回應是完整的 200，整頁職缺資料都在裡面。

原來，這個平台在某個時間點悄悄改成了純前端的單頁應用。伺服器現在送出的是一個空殼；真正的職缺是前端用 JS 再發一支 XHR 去打 API、拿到資料後才渲染出來的。在本機，Playwright 開的是完整的 Chromium，會執行那段 JavaScript、那支 XHR 跑得通，職缺正常渲染。在 runner 上，同一支 XHR——由一個坐在機房 IP 上的 headless 瀏覽器發出——被反爬蟲機制悄悄擋掉了。但對同一支 API 發一個普通的 HTTP request，從同一個機房 IP 出發，卻完全沒有問題。

這個區分非常關鍵：故障的本質是渲染被擋，不是IP 或地理位置被封。如果把它誤判成地理封鎖，就會去找 proxy 或換一個地區的 VPS——方向完全錯了。

真正的修法——把瀏覽器整個拿掉：既然 API 本身從機房 IP 打都打得通，最乾淨的解法就是不再驅動瀏覽器。scraper 現在直接對平台的 JSON API 發一個普通的 HTTP request，把回應欄位對映回整條 pipeline 原本就在用的內部職缺格式。附帶好處馬上就出現了：CI 不用再裝 Chromium、沒有 headless 瀏覽器指紋會被偵測、薪資和日期欄位從 API 拿到的，比從 DOM 刮出來的乾淨完整得多。

這裡有個特別的諷刺。這個修法，正好走回當初架構決策明確放棄的那條路——直接打平台的 API。幾個月前「一律 403」的結論，到頭來早就不成立了：一個普通的 user agent 加 Referer，不需要 cookie，就能直接打通。當年那個感覺很聰明、繞過平台防禦的決定，到這個 bug 浮現的時候，已經變成故障真正的元兇——而它當初想避開的那條路，反而成了最終取代它的解法。

真正救了這次 debug 的是什麼

這個 bug 是在調查中途接手的，用的是另一種跟 Claude 協作的模式——在對話視窗裡來回貼 log 很慢，每次「試試這個」都要切到終端機、複製輸出、貼回對話，一個來回三到五分鐘，而且完全看不到本機環境的真實狀況。直接在專案裡工作，讓 Claude 自己跑 scraper、自己讀輸出、立刻改檔案，把這個迴圈壓縮到幾秒鐘。但這次真正關鍵的，其實不是速度。

關鍵是兩件事。第一：能在真正會壞的環境裡量測，而不是只量一個替代品。Claude Code 本身也是跑在本機——它也不是 GitHub Actions 的 runner——但它可以寫一支診斷腳本、接進一支拋棄式 workflow，讓那支 workflow 真正在 runner 上執行，再把結果讀回來。正是這一步，才把「本機沒事」跟「雲端壞了」之間的差距，從猜測變成了測量。

第二件事，也更重要：第二幕修完之後，看起來已經修好了，當下很容易就此收尾。攔住這件事的，是一句懷疑：「你確定 API 在雲端真的打得通嗎？會不會跟本機又不一樣？」就是這句反問，逼出了「先在真正的 runner 上把 API 量一次，再決定要不要重寫」這一步。沒有這句話，這很可能會以另一次「修好了」收場，但其實仍然悄悄壞著。跟 AI 協作得好，不只是看它執行得多快——也在於人有沒有在正確的時刻踩一腳煞車。

學到的教訓

綠勾勾不代表健康。真正重要的是實際抓到的筆數，而「連續 N 天零筆」本身就該觸發主動告警——靜默失敗才是最危險的故障。

別只在跑得通的那台機器上診斷。本機（住宅 IP、完整瀏覽器）和雲端（機房 IP、headless）的行為，在關鍵的地方就是不一樣。一個只在本機驗證過的修法，等於還沒有在真正重要的地方被驗證過。

動手修之前，先分清根因的類別。「渲染被擋」和「IP 被封」從外面看起來幾乎一樣，但解法完全不同。花幾分鐘做一支探針把兩者分開，比猜錯方向重寫兩次省時間。

頁面是 SPA 的話，先去找它背後的 API。直接打 API，幾乎永遠比驅動一個完整瀏覽器去渲染同一份資料更快、更穩，也更不容易被反爬蟲盯上。

你最得意的那個決定，很可能就是日後要親手推翻的那個。用 headless 瀏覽器繞過反爬蟲，在當時要解決的那個問題上，是正確的決定。它只是沒有「永遠正確」。

最終成果

系統現在每天早上自動跑，無人值守。一次執行通常抓到約 200 筆原始職缺，評分過濾後一般剩 30 到 60 筆過門檻。拿掉瀏覽器之後，單次執行時間也從幾分鐘（大部分時間在裝 Chromium）降到幾十秒。GitHub Actions 的用量遠低於免費額度，Gmail SMTP 免費寄信。每月維護成本：零元。
